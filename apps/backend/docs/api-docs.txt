수강신청 기반 취약점 학습 서비스 - 1차 MVP API 상세 문서
=========================================================

목적
----
이 문서는 1차 MVP에서 웹이 자연스럽게 노출할 API와 취약점 표면을 정리한다.
웹은 일부러 만든 실습용 API보다 실제 수강신청/관리자 기능처럼 보이는 허술한 구현을 우선한다.
MCP는 이 허술한 기능 표면을 분석하여 OWASP Top 10:2025 취약점을 발견하는 흐름으로 맞춘다.

수정·설정 변경과 삭제 요청은 DELETE/PATCH 대신 POST로 통일한다.

팀원용 요약
----------
- API별 취약점·페이로드·**구현 여부** 한눈에 보기: `docs/vulnerability-matrix.md`
- OWASP별 API 목록: `docs/api-list.txt`
- 프론트 연동·가드 유무: `docs/api-docs-for-fe.txt`

구현 상태 (인증·SQLi)
--------------------
- **JWT 가드(`JwtAuthGuard`)가 붙은 API:** `POST /api/auth/logout`, `GET /api/auth/me` 만.
- 그 외 courses / enrollments / users / files / admin 은 문서에 `Authorization`이 적혀 있어도 **서버에서 토큰을 강제하지 않음** (실습·Burp 시 토큰 없이도 호출 가능).
- **SQL Injection:** `auth/login` 및 users / enrollments / courses / files / admin role 변경 등은 `repo.query()` + 문자열 결합으로 **구현됨**.
- **회원가입 SQLi:** 문서 시나리오 일부는 **미구현**(TypeORM `findOne`). 대신 body `role=admin` **권한 상승 가입**은 구현됨.

DB 스키마 기준
--------------
요청/응답 데이터는 현재 엔티티에 존재하는 필드를 기준으로 작성한다.

- `users`: `id`, `username`, `passwordHash`, `role`, `createdAt`, `recentLoginDate`, `status`
- `professors`: `id`, `userId`, `name`, `departmentName`
- `courses` (DB 컬럼): `id`, `code`, `name`, `description`, `professorId`, `category`, `lectureTime`, `classroom`, `credits`, `maxCapacity`, `currentCount`, `isVisible`, `createdAt`
- `courses` API 응답: DB `name` → `courseTitle`, `professors.name` join → `professorName`
- `enrollments`: `id`, `userId`, `courseId`, `status`, `enrolledAt`, `droppedAt`
- `files`: `id`, `courseId`, `uploaderId`, `originalName`, `storedPath`, `mimeType`, `fileSize`, `isPublic`, `uploadedAt`
- `refresh_tokens`: `id`, `userId`, `tokenHash`, `issuedAt`, `expiresAt`, `revokedAt`, `deviceInfo`
- `audit_logs`: `id`, `userId`, `action`, `resourceType`, `resourceId`, `ipAddress`, `requestMethod`, `requestPath`, `responseStatus`, `createdAt`
- `error_logs`: `id`, `errorType`, `message`, `stackTrace`, `requestPath`, `requestMethod`, `requestBody`, `userId`, `createdAt`
- `debug_configs`: `id`, `configKey`, `configValue`, `isSensitive`, `description`, `updatedAt`, `updatedBy`
- `cors_policies`(선택/향후): 1차 MVP에서는 관리 API를 두지 않는다. DB에는 행을 둘 수 있지만 노출 표면은 다른 misconfiguration 경로로 다룬다.

비영속 응답 예외
----------------
`accessToken`, `refreshToken`, `dependencies`, `buildInfo`처럼 DB 테이블에 저장되지 않는 값은 API 동작상 필요한 경우에만 "비영속 응답"으로 표시한다.

OWASP Top 10:2025 최종 점검 (현재 Nest HTTP 구현 기준)
------------------------------------------------------
- A01 Broken Access Control: **구현됨** — users/enrollments/files/admin IDOR·권한 우회.
- A02 Security Misconfiguration: **부분** — 파일 `storedPath` 등 내부 경로 노출. debug/error-log HTTP API는 **미구현**.
- A03 Software Supply Chain Failures: **미구현 HTTP** — `/api/system/*` 없음. 문서·DB 참고용만.
- A04 Cryptographic Failures: **부분** — JWT/refresh·파일 메타 노출. `/api/debug/secrets` **미구현**.
- A05 Injection: **구현됨** — login, courses(목록·상세·visibility), enrollments 전반, users, files, admin role. register SQLi·audit-log injection HTTP **미구현**.
- A06 Insecure Design: **구현됨** — 수강 정원/중복 미검증, visibility 권한 없음.
- A07 Authentication Failures: **구현됨** — 약한 가입 정책, `role=admin` 가입, 로그인 무제한 시도. refresh/logout 이슈는 시나리오별 확인.
- A08 Software or Data Integrity Failures: **부분** — 파일 삭제 API 구현. debug config 변조 HTTP **미구현**.
- A09 Security Logging and Alerting Failures: **미구현 HTTP** — audit/error-log API 없음. DB 테이블만 유지.
- A10 Mishandling of Exceptional Conditions: **부분** — 잘못된 path/query 시 DB 에러·stack trace 노출(케이스별).

[API]
POST /api/auth/register

[기능]
- 회원가입
- 기본 role은 `student`로 생성한다.
- 기본 status는 `active`로 생성한다.
- 토큰 발급 시 `users.recentLoginDate`를 현재 시각으로 갱신

[요청데이터]
- 헤더
  - `Content-Type: application/json`
- 바디
  - `id?`: DTO에만 존재. **현재 create 로직에서 무시** (SQLi 미사용)
  - `username`: `users.username`
  - `passwordHash`: `users.passwordHash` 생성에 사용되는 원문 비밀번호 입력값
  - `role?`: 생략 시 `student`. **`admin` / `professor` 직접 전달 가능**
- 공격이 들어오는 방법
  - `role=admin` 전달로 **관리자 계정 생성**(권한 상승)
  - `username`·`passwordHash` 극단적으로 긴 값 → DB/validation 에러·`stackTrace` 유도
  - 비정상 JSON으로 예외 유도
- (참고·미구현) 설계 문서에 있던 회원가입 SQLi(`username`에 `' OR '1'='1` 등)는 **현재 TypeORM `findOne` 중복 확인**이라 재현되지 않음. SQLi 실습은 `POST /api/auth/login` 등 **구현된 raw SQL API** 사용.

[응답데이터]
- 정상 응답
  - `id`, `username`, `role`, `createdAt`
- 취약점 노출 응답
  - 비밀번호 복잡도/길이와 무관하게 계정 생성 성공
  - 사용자가 전달한 `role=admin`이 그대로 반영된 응답
  - 중복 `username` 시 409. 비정상 입력 시 `message`, `stackTrace`, `requestBody`가 포함된 에러 응답 가능

[연결 UI]
- 회원가입 페이지

[노출 취약점]
- A07: 비밀번호 정책 없음
- A07: body `role` 조작으로 admin/professor 가입
- A01: (간접) 가입 직후 토큰으로 admin API 접근 시도
- A10: 비정상 입력·중복 가입 시 예외 정보 노출

[API]
POST /api/auth/login

[기능]
- 로그인
- JWT access token 발급
- refresh token 발급 및 `refresh_tokens` 저장
- 토큰 발급 시 `users.recentLoginDate`를 현재 시각으로 갱신

[요청데이터]
- 헤더
  - `Content-Type: application/json`
- 바디
  - `username`: `users.username`
  - `passwordHash`: `users.passwordHash` 비교에 사용되는 원문 비밀번호 입력값
- 공격이 들어오는 방법
  - 동일 계정에 대한 반복 로그인 시도
  - `username=admin' --` 형태로 비밀번호 조건 우회
  - `username=admin' OR '1'='1' --` 형태로 첫 번째 사용자 인증 유도
  - `username=admin' AND LEFT(password_hash,1)='a' --` 형태의 blind SQL Injection으로 `passwordHash` 일부 추론
  - `username` 50자 초과 또는 비밀번호 255자 초과 수준의 긴 문자열로 DB/비교 로직 에러 유도
  - malformed JSON body 전송
  - `X-Test-Delay: true` 헤더로 timeout 처리 확인

[응답데이터]
- 정상 응답
  - 사용자: `id`, `username`, `role`, `createdAt`
  - 비영속 응답: `accessToken`, `refreshToken`
- 취약점 노출 응답
  - 반복 로그인에도 차단/지연/429 응답이 동작하지 않음
  - SQLi payload로 비밀번호 비교를 우회해 로그인 성공
  - `LEFT(password_hash,1)` 조건 결과에 따라 401/500/응답 시간 차이가 발생
  - 길이 초과 입력 시 컬럼명, 제한 길이, DB 에러 메시지, `stackTrace` 노출
  - refresh token이 장기간 유효하거나 재사용 가능
  - 에러 응답에 `passwordHash` 비교 실패 사유, `stackTrace`, `requestBody` 노출

[연결 UI]
- 로그인 / 회원가입 페이지

[노출 취약점]
- A07: 로그인 시도 제한 없음
- A04: JWT/refresh token 취약 구성
- A05: 로그인 SQL Injection
- A10: malformed input 또는 timeout 처리 실패

[연결 UI]

[API]
GET /api/courses

[기능]
- 강의 목록 조회
- 검색 및 카테고리 필터링
- 기본: `is_visible = true` 공개 강의만 반환

[요청데이터]
- 헤더
  - `Authorization: Bearer <accessToken>`
- 바디
  - 없음
- Query
  - `keyword`: `courses.name`, `courses.code`, `courses.description` 검색에 사용
  - `category`: `courses.category`
  - `isVisible`: 생략 시 공개만; 명시 시 해당 값으로 필터(실습: `false`로 숨김 강의 노출)
- 공격이 들어오는 방법
  - `keyword=' OR '1'='1`
  - `category=' OR is_visible=false --`
  - `isVisible=false` 직접 전달로 숨김 강의 조회
  - `q`, `keyword`에 `@@`, `1/0`, `<script>` 입력으로 예외 유도

[응답데이터]
- 정상 응답
  - 목록 항목: `id`, `code`, `courseTitle`, `description`, `professorId`, `professorName`, `category`, `lectureTime`, `classroom`, `credits`, `maxCapacity`, `currentCount`, `isVisible`, `createdAt`
- 취약점 노출 응답
  - `isVisible=false` 등 쿼리로 숨김 강의까지 반환
  - Injection payload로 필터 우회·전체 강의 반환
  - 에러 응답에 raw SQL, `Error:`, `stackTrace` 노출

[연결 UI]
- 강의 목록 페이지

[노출 취약점]
- A05: SQL Injection
- A01: 숨김 강의 접근 제어 실패
- A10: 검색 조건 예외 처리 실패

  - `status`: `enrollments.status`, 기본값 `enrolled`
- 공격이 들어오는 방법
  - `userId`를 다른 사용자의 `users.id`로 변경
  - 프론트에서 신청 버튼이 비활성화된 정원 마감 강의의 `courseId`를 직접 요청하여 서버 측 정원 검증 우회
  - 정원 마감 강의에 동시 요청 전송
  - 같은 `courseId`를 반복 전송
  - `status` 필드에 SQL 조각 삽입 (INSERT 문자열 결합)
  - `X-Test-Delay: true` 헤더로 timeout 처리 확인

[응답데이터]
- 정상 응답
  - `id`, `userId`, `courseId`, `status`, `enrolledAt`, `droppedAt`
  - 관련 강의 상태: `id`, `maxCapacity`, `currentCount`
- 취약점 노출 응답
  - 다른 `userId`로 수강신청 성공
  - `currentCount >= maxCapacity`인 강의에도 수강신청 성공
  - `currentCount > maxCapacity` 상태의 강의 반환
  - 중복 신청 DB constraint 에러, raw error, `stackTrace` 노출
  - timeout 상황에서 500 또는 응답 지연

[연결 UI]

- A10 예외 처리 취약점 탐지 흐름

[노출 취약점]
- A06: 프론트 상태에만 의존한 서버 측 정원 검증 부재
- A06: race condition으로 정원 초과
- A06: 중복 신청/신청 정책 우회

[구현 제거 안내] (감사·에러 로그 / 보안 이벤트 / 시스템·디버그 진단 API)

아래에 해당하는 HTTP API는 현재 Nest 백엔드 구현에서 **제거**되었다.
`audit_logs`, `error_logs`, `debug_configs` 등 **DB 테이블·엔티티는 유지**하며, 향후 동일 기능을 다시 노출할 때를 대비한 스키마/문서 참고용으로만 둔다.

제거된 엔드포인트(이전 사양, 참고용 목록):
- GET  /api/admin/audit-logs
- POST /api/admin/audit-logs/clear
- GET  /api/admin/error-logs
- GET  /api/security/events
- GET  /api/system/dependencies
- GET  /api/system/build-info
- GET  /api/debug/configs
- POST /api/debug/configs/:key
- GET  /api/debug/secrets
- POST /api/debug/query-test

상세 요청·응답·공격 시나리오는 Git 히스토리 또는 백업본의 본문(구 `api-docs.txt` 674~1022행 부근)을 참고한다.
